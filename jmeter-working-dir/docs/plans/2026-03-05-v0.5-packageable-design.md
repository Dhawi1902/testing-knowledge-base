## CLI Interface

Replace raw `sys.argv` parsing in `__main__.py` with argparse subcommands.

```
loadlitmus                    # default: starts the dashboard (same as 'serve')
loadlitmus serve              # start the dashboard
loadlitmus serve --port 9000  # override port
loadlitmus serve --host 0.0.0.0  # override host
loadlitmus init               # scaffold project in current directory
loadlitmus init <path>        # scaffold project in a specific directory
loadlitmus --version          # print "LoadLitmus 0.5.0" and exit
loadlitmus --help             # show available commands
```

**Framework:** argparse (stdlib). No new dependencies. Migrate to click in a future version if the CLI grows significantly — the migration is a syntax swap in one file (~30 min).

**Default behavior:** Running `loadlitmus` with no arguments starts the dashboard. This means double-clicking `loadlitmus.exe` launches the server immediately — the most intuitive behavior for non-CLI users.

## Project Init

`loadlitmus init` scaffolds a new performance testing project directory. It creates the folder structure only — the existing first-run setup wizard in the web UI handles JMeter path detection, settings, and configuration.

**Created structure:**

```
<project-dir>/
├── config/
│   └── vm_config.json        # empty template with comments
├── test_plan/
├── test_data/
├── results/
├── slaves.txt                # empty JSON array: []
└── config.properties         # minimal template with comments
```

**Behavior:**
- If `<path>` is omitted, uses the current working directory
- If the directory already contains project files, warn and skip (don't overwrite)
- Print what was created and suggest running `loadlitmus serve` next

**Note:** When running from exe, the `webapp/` directory is bundled inside the executable — `init` does NOT extract webapp files into the project directory.

## Version Management

Single source of truth in `webapp/__version__.py`:

```python
__version__ = "0.5.0"
```

Used by:
- `loadlitmus --version` — reads and prints it
- `pyproject.toml` — references it via `dynamic = ["version"]`
- `main.py` — available for the `/api/version` endpoint (planned in v0.8)
- `scripts/build.py` — stamps the exe with the version
- PyInstaller — embeds in exe metadata

## Dependency Split

Move from `requirements.txt` to `pyproject.toml` with separated dependency groups.

**`pyproject.toml`:**

```toml
[build-system]
requires = ["setuptools>=68.0"]
build-backend = "setuptools.backends._legacy:_Backend"

[project]
name = "loadlitmus"
dynamic = ["version"]
description = "Web dashboard for managing performance test projects"
requires-python = ">=3.11"
dependencies = [
    "fastapi>=0.104.0",
    "uvicorn[standard]>=0.24.0",
    "jinja2>=3.1.0",
    "python-multipart>=0.0.6",
    "websockets>=12.0",
    "aiofiles>=23.0",
    "pandas>=2.0",
    "numpy>=1.24",
    "httpx>=0.25.0",
    "paramiko>=3.4",
    "scp>=0.14",
]

[project.optional-dependencies]
dev = [
    "pytest>=7.0",
    "pytest-cov>=4.0",
    "pyinstaller>=6.0",
]

[project.scripts]
loadlitmus = "webapp.__main__:main"

[tool.setuptools.dynamic]
version = {attr = "webapp.__version__.__version__"}
```

**`requirements.txt`** stays but with runtime deps only (for users who prefer `pip install -r`). Add a comment pointing to `pyproject.toml` for dev deps.

**Install commands:**
- `pip install .` — runtime only
- `pip install ".[dev]"` — runtime + test + build tools

## PyInstaller Packaging

**Mode:** One-file (`--onefile`). Produces a single `loadlitmus.exe`. Clean to distribute — drop into any project directory and run.

**Spec file (`loadlitmus.spec`):**

Declares which non-Python files to bundle:

```python
datas = [
    ('webapp/templates', 'templates'),
    ('webapp/static', 'static'),
    ('webapp/config', 'config'),
    ('webapp/prompts', 'prompts'),
    ('webapp/jtl_filter.py', '.'),
]
```

**Runtime path detection:**

When PyInstaller builds a one-file exe, it extracts bundled files to a temporary directory at runtime. The app needs to know where its assets are:

```python
import sys
from pathlib import Path

def get_app_dir() -> Path:
    if getattr(sys, 'frozen', False):
        return Path(sys._MEIPASS)
    return Path(__file__).resolve().parent
```

This replaces the current `APP_DIR = Path(__file__).resolve().parent` in `main.py`.

## How PyInstaller Works (Reference)

This section explains the key PyInstaller concepts for future reference.

### What PyInstaller Does

PyInstaller analyzes your Python entry point (`__main__.py`), traces all `import` statements recursively, and bundles everything into a distributable package. The result is an executable that includes:

1. **A Python interpreter** — the exact Python version you built with
2. **All imported Python packages** — fastapi, uvicorn, pandas, etc.
3. **Non-Python data files** — templates, CSS, JS, config (declared in the `.spec` file)
4. **A bootloader** — small C program that starts everything up

The end user does NOT need Python installed. The exe is self-contained.

### One-File vs One-Dir

| Mode | Output | Startup | Distribution |
|------|--------|---------|-------------|
| `--onefile` | Single `loadlitmus.exe` (~50-80 MB) | ~1-2s (extracts to temp) | Drop one file anywhere |
| `--onedir` | `dist/loadlitmus/` folder with many files | Instant | Must keep folder intact |

We use **one-file** because distribution simplicity matters more than startup speed for a web server that runs for hours.

### The `frozen` Flag

When running inside a PyInstaller bundle, Python sets `sys.frozen = True`. This is how you detect whether you're running from source or from an exe:

```python
if getattr(sys, 'frozen', False):
    # Running from PyInstaller exe
    # sys._MEIPASS = temp directory where files are extracted
else:
    # Running from source (python -m webapp)
    # __file__ = actual file path on disk
```

### `sys._MEIPASS`

In one-file mode, PyInstaller extracts all bundled files to a temporary directory when the exe starts. `sys._MEIPASS` is the path to that directory. Example:

```
C:\Users\user\AppData\Local\Temp\_MEI12345\
├── templates/        # your Jinja2 templates
├── static/           # your CSS, JS, images
├── config/           # your config files
└── ... (Python stdlib, packages, etc.)
```

This directory is automatically cleaned up when the exe exits.

### The `.spec` File

The `.spec` file is a Python script that tells PyInstaller what to bundle. Key sections:

```python
# Analysis — what to trace
a = Analysis(
    ['webapp/__main__.py'],    # entry point
    datas=[...],               # non-Python files to include
    hiddenimports=[...],       # imports PyInstaller can't detect automatically
)

# PYZ — compressed Python modules
pyz = PYZ(a.pure)

# EXE — the final executable
exe = EXE(
    pyz, a.scripts, a.binaries, a.datas,
    name='loadlitmus',         # output filename
    onefile=True,
    console=True,              # show terminal window (needed for CLI)
    icon='webapp/static/favicon.ico',  # optional exe icon
)
```

### Hidden Imports

PyInstaller traces `import` statements statically. If your code uses dynamic imports (e.g., `importlib.import_module()`, plugin loading), PyInstaller won't find them. You declare these as `hiddenimports` in the spec file:

```python
hiddenimports = ['uvicorn.logging', 'uvicorn.loops.auto', 'uvicorn.protocols.http.auto']
```

FastAPI + uvicorn commonly need hidden imports because uvicorn loads protocol handlers dynamically.

### Common Gotchas

1. **Path assumptions** — Code that uses `Path(__file__)` to find templates/static will break in exe mode because `__file__` doesn't exist. Always use the `get_app_dir()` helper.

2. **Data files not included** — PyInstaller only bundles what you declare in `datas`. If you add a new `webapp/` subdirectory later, you must update the spec file.

3. **Large exe size** — pandas + numpy + uvicorn make the exe ~50-80 MB. This is normal. The entire Python runtime is inside.

4. **Antivirus false positives** — PyInstaller exes sometimes trigger antivirus warnings because the extraction-to-temp pattern looks suspicious. This is a known issue with no fix other than code-signing (out of scope for v0.5).

5. **Console window** — We build with `console=True` so the terminal stays open (needed to see server logs and Ctrl+C to stop). A `--no-console` option could be added later for background service mode.

## Build Script

`scripts/build.py` — automates the build and validates the output.

**Steps:**
1. Read version from `webapp/__version__.py`
2. Run PyInstaller with the spec file
3. Verify `dist/loadlitmus.exe` exists
4. **Smoke test:**
   - Start the exe as a subprocess
   - Wait for it to bind the port (poll with HTTP GET)
   - Verify GET `/` returns HTTP 200
   - Kill the process
5. Print build summary (version, file size, path)

**Usage:**
```bash
python scripts/build.py
```

**Output:**
```
dist/loadlitmus.exe
```

### New Files

| File | Purpose |
|------|---------|
| `webapp/__version__.py` | `__version__ = "0.5.0"` |
| `pyproject.toml` | Package metadata, entry point, dep split |
| `loadlitmus.spec` | PyInstaller bundling configuration |
| `scripts/build.py` | Build automation + smoke test |

### Modified Files

| File | Change |
|------|--------|
| `webapp/__main__.py` | Replace `sys.argv` parsing with argparse (`serve`, `init`, `--version`). Extract `main()` function as entry point. |
| `webapp/main.py` | Replace `APP_DIR = Path(__file__).resolve().parent` with `get_app_dir()` helper for frozen exe support |
| `webapp/requirements.txt` | Runtime deps only. Comment pointing to `pyproject.toml` for dev deps. |
| `.gitignore` | Add `dist/`, `build/`, `*.spec` artifacts |

### Not Changed

| File | Why |
|------|-----|
| `webapp/project.json` | Auto-generated at runtime, no changes needed |
| `webapp/settings.json` | Auto-generated at runtime, no changes needed |
| CI workflow | v0.5 doesn't add CI builds (that's v0.9) |
